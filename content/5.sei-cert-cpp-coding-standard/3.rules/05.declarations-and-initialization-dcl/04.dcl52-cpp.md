# DCL52-CPP. Never qualify a reference type with const or volatile

C++ does not allow you to change the value of a reference type, effectively treating all references as being `const` qualified. The C++ Standard, \[dcl.ref\], paragraph 1 \[ [ISO/IEC 14882-2014](/sei-cert-cpp-coding-standard/back-matter/aa-bibliography#AA.Bibliography-ISO/IEC14882-2014) \], states the following:

> Cv-qualified references are ill-formed except when the cv-qualifiers are introduced through the use of a *typedef-name* (7.1.3, 14.1) or *decltype-specifier* (7.1.6.2), in which case the cv-qualifiers are ignored.

Thus, C++ prohibits or ignores the [cv-qualification](/sei-cert-cpp-coding-standard/back-matter/bb-definitions#BB.Definitions-cvqualify) of a reference type. Only a value of non-reference type may be cv-qualified.

When attempting to `const` -qualify a type as part of a declaration that uses reference type, a programmer may accidentally write

::code-block{quality="bad"}
``` cpp
char &const p;
```
::

instead of

::code-block{quality="good"}
``` cpp
char const &p; // Or: const char &p;
```
::

Do not attempt to cv-qualify a reference type because it results in [undefined behavior](/sei-cert-cpp-coding-standard/back-matter/bb-definitions#BB.Definitions-undefinedbehavior) . A [conforming](/sei-cert-cpp-coding-standard/back-matter/bb-definitions#BB.Definitions-conformingprogram) compiler is required to issue a [diagnostic message](/sei-cert-cpp-coding-standard/back-matter/bb-definitions#BB.Definitions-diagnosticmessage) . However, if the compiler does not emit a [fatal diagnostic](/sei-cert-cpp-coding-standard/back-matter/bb-definitions#BB.Definitions-fataldiagnostic) , the program may produce surprising results, such as allowing the character referenced by `p` to be mutated.

## Noncompliant Code Example

In this noncompliant code example, a `const` -qualified reference to a `char` is formed instead of a reference to a `const` -qualified `char.` This results in undefined behavior.

::code-block{quality="bad"}
``` cpp
#include <iostream>
 
void f(char c) {
  char &const p = c;
  p = 'p';
  std::cout << c << std::endl;
}
```
::

### Implementation Details (MSVC)

With [Microsoft Visual Studio](/sei-cert-cpp-coding-standard/back-matter/bb-definitions#BB.Definitions-msvc) 2015, this code compiles successfully with a warning diagnostic.

``` java
warning C4227: anachronism used : qualifiers on reference are ignored
```

When run, the code outputs the following.

``` java
p
```

### Implementation Details (Clang)

With [Clang](/sei-cert-cpp-coding-standard/back-matter/bb-definitions#BB.Definitions-clang) 3.9, this code produces a fatal diagnostic.

``` java
error: 'const' qualifier may not be applied to a reference
```

## Noncompliant Code Example

This noncompliant code example correctly declares `p` to be a reference to a const-qualified `char` . The subsequent modification of `p` makes the program [ill-formed](/sei-cert-cpp-coding-standard/back-matter/bb-definitions#BB.Definitions-ill-formed) .

::code-block{quality="bad"}
``` cpp
#include <iostream>
 
void f(char c) {
  const char &p = c;
  p = 'p'; // Error: read-only variable is not assignable
  std::cout << c << std::endl;
}
```
::

## Compliant Solution

This compliant solution removes the `const` qualifier.

::code-block{quality="good"}
``` cpp
#include <iostream>
 
void f(char c) {
  char &p = c;
  p = 'p';
  std::cout << c << std::endl;
}
```
::

## Risk Assessment

A `const` or `volatile` reference type may result in undefined behavior instead of a fatal diagnostic, causing unexpected values to be stored and leading to possible data integrity violations.
