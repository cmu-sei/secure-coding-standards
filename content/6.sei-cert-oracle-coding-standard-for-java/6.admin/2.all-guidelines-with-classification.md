# All Guidelines with Classification

## Guidelines

## Security

This chapter contains guidelines that are concerned specifically with ensuring security of Java-based applications. They are concerned with issues such as

1.  Dealing with sensitive data.
2.  Avoiding common injection attacks.
3.  Language features that can be misused to compromise security.
4.  Details of Java's fine-grained security mechanism.

## Defensive Programming

This chapter contains guidelines for defensive programming. Defensive programming is carefully guarded programming that helps you to construct reliable software by designing each component to protect itself as much as possible, for example, by checking that undocumented assumptions remain valid \[ [Goodliffe 2006](/sei-cert-oracle-coding-standard-for-java/back-matter/rec-aa-references#Rec.AA.References-Goodliffe06) \] . These guidelines address areas of the Java language that can help to constrain the effect of an error or help to recover from an error.  

-   [CON52-J. Document thread-safety and use annotations where applicable](/sei-cert-oracle-coding-standard-for-java/recommendations/concurrency-con/con52-j)
-   [DCL53-J. Minimize the scope of variables](/sei-cert-oracle-coding-standard-for-java/recommendations/declarations-and-initialization-dcl/dcl53-j)
-   [DCL56-J. Do not attach significance to the ordinal associated with an enum](/sei-cert-oracle-coding-standard-for-java/recommendations/declarations-and-initialization-dcl/dcl56-j)
-   [DCL58-J. Enable compile-time type checking of variable arity parameter types](/sei-cert-oracle-coding-standard-for-java/recommendations/declarations-and-initialization-dcl/dcl58-j)
-   [DCL59-J. Do not apply public final to constants whose value might change in later releases](/sei-cert-oracle-coding-standard-for-java/recommendations/declarations-and-initialization-dcl/dcl59-j)
-   [DCL60-J. Avoid cyclic dependencies between packages](/sei-cert-oracle-coding-standard-for-java/recommendations/declarations-and-initialization-dcl/dcl60-j)
-   [ERR51-J. Prefer user-defined exceptions over more general exception types](/sei-cert-oracle-coding-standard-for-java/recommendations/exceptional-behavior-err/err51-j)
-   [ERR53-J. Try to gracefully recover from system errors](/sei-cert-oracle-coding-standard-for-java/recommendations/exceptional-behavior-err/err53-j)

## Best Practices

This chapter includes guidelines that

1.  Are essential for ensuring correctness of Java code.
2.  Contain specific, implementation-level Java coding advice.
3.  Have a higher normative requirement than the defensive programming category.

The guidelines deal with generally low-level Java language features that can easily be misused by the unwary.

## Program Understandability

Program understandability is the ease with which the program can be understood—that is, the ability to determine what a program does and how it works by reading its source code and accompanying documentation \[ [Grubb 2013](/sei-cert-oracle-coding-standard-for-java/back-matter/rec-aa-references#Rec.AA.References-Grubb13) \]. Another term used to describe this property of source code is *readability* . Understandable code is easier to maintain because software maintainers are less likely to introduce defects into code that is clear and comprehensible. Understandability helps in manual analysis of source code because it allows the auditor to more easily spot defects and vulnerabilities.

The guidelines in this chapter are mainly stylistic in nature; they will help a Java programmer to write clearer, more readable code.

-   [DCL50-J. Use visually distinct identifiers](/sei-cert-oracle-coding-standard-for-java/recommendations/declarations-and-initialization-dcl/dcl50-j)
-   [DCL57-J. Avoid ambiguous overloading of variable arity methods](/sei-cert-oracle-coding-standard-for-java/recommendations/declarations-and-initialization-dcl/dcl57-j)
-   [ERR52-J. Avoid in-band error indicators](/sei-cert-oracle-coding-standard-for-java/recommendations/exceptional-behavior-err/err52-j)
-   [EXP51-J. Do not perform assignments in conditional expressions](/sei-cert-oracle-coding-standard-for-java/recommendations/expressions-exp/exp51-j)
-   [EXP52-J. Use braces for the body of an if, for, or while statement](/sei-cert-oracle-coding-standard-for-java/recommendations/expressions-exp/exp52-j)
-   [EXP53-J. Use parentheses for precedence of operation](/sei-cert-oracle-coding-standard-for-java/recommendations/expressions-exp/exp53-j)
-   [FIO50-J. Do not make assumptions about file creation](/sei-cert-oracle-coding-standard-for-java/recommendations/input-output-fio/fio50-j)
-   [MET53-J. Ensure that the clone() method calls super.clone()](/sei-cert-oracle-coding-standard-for-java/recommendations/methods-met/met53-j)
-   [MSC51-J. Do not place a semicolon immediately following an if, for, or while condition](/sei-cert-oracle-coding-standard-for-java/recommendations/miscellaneous-msc/msc51-j)
-   [MSC52-J. Finish every set of statements associated with a case label with a break statement](/sei-cert-oracle-coding-standard-for-java/recommendations/miscellaneous-msc/msc52-j)
-   [MSC54-J. Avoid inadvertent wrapping of loop counters](/sei-cert-oracle-coding-standard-for-java/recommendations/miscellaneous-msc/msc54-j)
-   [MSC55-J. Use comments consistently and in a readable fashion](/sei-cert-oracle-coding-standard-for-java/recommendations/miscellaneous-msc/msc55-j)

## Programmer Misconceptions

This chapter contains guidelines that address

1.  Misconceptions about Java APIs and language features.
2.  Assumptions and ambiguity-laced programs.
3.  Situations in which the programmer wanted to do *x* but ended up doing *y.*

They address areas where developers often make unwarranted assumptions about how Java works or where ambiguities can easily be introduced.

-   [CON50-J. Do not assume that declaring a reference volatile guarantees safe publication of the members of the referenced object](/sei-cert-oracle-coding-standard-for-java/recommendations/concurrency-con/con50-j)
-   [CON51-J. Do not assume that the sleep(), yield(), or getState() methods provide synchronization semantics](/sei-cert-oracle-coding-standard-for-java/recommendations/concurrency-con/con51-j)
-   [EXP50-J. Do not confuse abstract object equality with reference equality](/sei-cert-oracle-coding-standard-for-java/recommendations/expressions-exp/exp50-j)
-   [EXP54-J. Understand the differences between bitwise and logical operators](/sei-cert-oracle-coding-standard-for-java/recommendations/expressions-exp/exp54-j)
-   [FIO53-J. Use the serialization methods writeUnshared() and readUnshared() with care](/sei-cert-oracle-coding-standard-for-java/recommendations/input-output-fio/fio53-j)
-   [IDS55-J. Understand how escape characters are interpreted when strings are loaded](/sei-cert-oracle-coding-standard-for-java/recommendations/input-validation-and-data-sanitization-ids/ids55-j)
-   [MET50-J. Avoid ambiguous or confusing uses of overloading](/sei-cert-oracle-coding-standard-for-java/recommendations/methods-met/met50-j)
-   [MET51-J. Do not use overloaded methods to differentiate between runtime types](/sei-cert-oracle-coding-standard-for-java/recommendations/methods-met/met51-j)
-   [NUM51-J. Do not assume that the remainder operator always returns a nonnegative result for integral operands](/sei-cert-oracle-coding-standard-for-java/recommendations/numeric-types-and-operations-num/num51-j)
-   [OBJ50-J. Never confuse the immutability of a reference with that of the referenced object](/sei-cert-oracle-coding-standard-for-java/recommendations/object-orientation-obj/obj50-j)
-   [OBJ54-J. Do not attempt to help the garbage collector by setting local reference variables to null](/sei-cert-oracle-coding-standard-for-java/recommendations/object-orientation-obj/obj54-j)
