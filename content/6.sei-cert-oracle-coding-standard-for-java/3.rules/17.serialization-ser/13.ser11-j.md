# SER11-J. Prevent overwriting of externalizable objects

Classes that implement the `Externalizable` interface must provide the `readExternal()` and `writeExternal()` methods. These methods have package-private or public access, and so they can be called by trusted and untrusted code alike. Consequently, programs must ensure that these methods execute only when intended and that they cannot overwrite the internal state of objects at arbitrary points during program execution.

## Noncompliant Code Example

This noncompliant code example allows any caller to reset the value of the object at any time because the `readExternal()` method is necessarily declared to be public and lacks protection against hostile callers:

::code-block{quality="bad"}
``` java
public void readExternal(ObjectInput in) 
                         throws IOException, ClassNotFoundException {
   // Read instance fields
   this.name = (String) in.readObject();
   this.UID = in.readInt();
   // ...
}
```
::

## Compliant Solution

This compliant solution protects against multiple initialization through the use of a Boolean flag that is set after the instance fields have been populated. It also protects against [race conditions](/sei-cert-oracle-coding-standard-for-java/back-matter/rule-bb-glossary) by synchronizing on a private lock object (see [LCK00-J. Use private final lock objects to synchronize classes that may interact with untrusted code](/sei-cert-oracle-coding-standard-for-java/rules/locking-lck/lck00-j) ).

::code-block{quality="good"}
``` java
private final Object lock = new Object();
private boolean initialized = false;

public void readExternal(ObjectInput in)
                         throws IOException, ClassNotFoundException {
  synchronized (lock) {
    if (!initialized) {
      // Read instance fields
      this.name = (String) in.readObject();
      this.UID = in.readInt();
      // ...  
      initialized = true;
    } else {
      throw new IllegalStateException();
    }
  }
}
```
::

Note that this compliant solution is inadequate to protect [sensitive data](/sei-cert-oracle-coding-standard-for-java/back-matter/rule-bb-glossary#RuleBB.Glossary-sen) .

Failure to prevent the overwriting of an externalizable object can corrupt the state of the object.
