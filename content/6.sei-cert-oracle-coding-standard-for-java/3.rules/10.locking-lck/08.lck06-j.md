# LCK06-J. Do not use an instance lock to protect shared static data

Programs must not use instance locks to protect static shared data because instance locks are ineffective when two or more instances of the class are created. Consequently, failure to use a static lock object leaves the shared state unprotected against concurrent access. Lock objects for classes that can interact with [untrusted code](/sei-cert-oracle-coding-standard-for-java/back-matter/rule-bb-glossary#RuleBB.Glossary-untrustedcode) must also be private and final, as shown in [LCK00-J. Use private final lock objects to synchronize classes that may interact with untrusted code](/sei-cert-oracle-coding-standard-for-java/rules/locking-lck/lck00-j) .

## Noncompliant Code Example (Nonstatic Lock Object for Static Data)

This noncompliant code example attempts to guard access to the static `counter` field using a nonstatic lock object. When two `Runnable` tasks are started, they create two instances of the lock object and lock on each instance separately.

::code-block{quality="bad"}
``` java
public final class CountBoxes implements Runnable {
  private static volatile int counter;
  // ...
  private final Object lock = new Object();

  @Override public void run() {
    synchronized (lock) {
      counter++;
      // ...
    }
  }

  public static void main(String[] args) {
    for (int i = 0; i < 2; i++) {
    new Thread(new CountBoxes()).start();
    }
  }
}
```
::

This example fails to prevent either thread from observing an inconsistent value of `counter` because the increment operation on volatile fields fails to be atomic in the absence of proper [synchronization](/sei-cert-oracle-coding-standard-for-java/back-matter/rule-bb-glossary#RuleBB.Glossary-synchro) (see [VNA02-J. Ensure that compound operations on shared variables are atomic](/sei-cert-oracle-coding-standard-for-java/rules/visibility-and-atomicity-vna/vna02-j) for more information).

## Noncompliant Code Example (Method Synchronization for Static Data)

This noncompliant code example uses method synchronization to protect access to a static class `counter` field:

::code-block{quality="bad"}
``` java
public final class CountBoxes implements Runnable {
  private static volatile int counter;
  // ...

  public synchronized void run() {
    counter++;
    // ...
  }
  // ...
}
```
::

In this case, the method synchronization uses the intrinsic lock associated with each instance of the class rather than the intrinsic lock associated with the class itself. Consequently, threads constructed using different `Runnable` instances may observe inconsistent values of `counter` .

## Compliant Solution (Static Lock Object)

This compliant solution ensures the atomicity of the increment operation by locking on a static object:

::code-block{quality="good"}
``` java
public class CountBoxes implements Runnable {
  private static int counter;
  // ...
  private static final Object lock = new Object();

  public void run() {
    synchronized (lock) {
      counter++;
      // ...
    }
  }
  // ...
}
```
::

It is unnecessary to declare the `counter` variable volatile when using [synchronization](/sei-cert-oracle-coding-standard-for-java/back-matter/rule-bb-glossary#RuleBB.Glossary-synchro) .
