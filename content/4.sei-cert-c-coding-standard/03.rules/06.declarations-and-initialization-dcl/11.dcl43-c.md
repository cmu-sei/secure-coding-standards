# DCL43-C. Avoid undefined behavior when declaring variables with auto

C23 \[ [ISO/IEC 9899:2024](/sei-cert-c-coding-standard/back-matter/aa-bibliography#AA.Bibliography-ISO-IEC9899-2024) \] added support for type inference using the `auto` keyword.  This new feature comes with two forms of undefined behavior:

| Undefined <br> Behavior | Meaning |
| --- | --- |
| [77](/sei-cert-c-coding-standard/back-matter/cc-undefined-behavior#CC.UndefinedBehavior-ub_77) | A declaration for which a type is inferred contains a pointer, array, or function declarators |
| [78](/sei-cert-c-coding-standard/back-matter/cc-undefined-behavior#CC.UndefinedBehavior-ub_78) | A declaration for which a type is inferred contains no or more than one declarators |

## Noncompliant Code Example

This noncompliant code example declares two variables in one declaration using `auto`.

::code-block{quality="bad"}
``` c
auto i = 3, j = 4.5; // Undefined Behavior
```
::

## Compliant Solution

In this compliant solution, each declarator is sequestered to its own declaration.

::code-block{quality="good"}
``` c
auto i = 3;
auto j = 4.5;
```
::

## Risk Assessment

Using `auto` to declare zero or two or more variables is undefined behavior [78](/sei-cert-c-coding-standard/back-matter/cc-undefined-behavior#CC.UndefinedBehavior-ub_78). Using `auto` for declarations that contain a pointer, array, or function declarators is undefined behavior [77](/sei-cert-c-coding-standard/back-matter/cc-undefined-behavior#CC.UndefinedBehavior-ub_77).
