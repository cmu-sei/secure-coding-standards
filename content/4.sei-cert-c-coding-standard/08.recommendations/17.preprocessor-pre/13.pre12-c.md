# PRE12-C. Do not define unsafe macros

An [unsafe function-like macro](/sei-cert-c-coding-standard/back-matter/bb-definitions#BB.Definitions-unsafefunction-likemacro) is one that, when expanded, evaluates its argument more than once or does not evaluate it at all. Contrasted with function calls, which always evaluate each of their arguments exactly once, unsafe function-like macros often have unexpected and surprising effects and lead to subtle, hard-to-find defects (see [PRE31-C. Avoid side effects in arguments to unsafe macros](/sei-cert-c-coding-standard/rules/preprocessor-pre/pre31-c) ). Consequently, every [function-like macro](/sei-cert-c-coding-standard/back-matter/bb-definitions#BB.Definitions-function-likemacro) should evaluate each of its arguments exactly once. Alternatively and preferably, defining function-like macros should be avoided in favor of inline functions (see [PRE00-C. Prefer inline or static functions to function-like macros](/sei-cert-c-coding-standard/recommendations/preprocessor-pre/pre00-c) ).

<a id="PRE12C.Donotdefineunsafemacros-nce_multiple_evaluation"></a>

## Noncompliant Code Example (Multiple Argument Evaluation)

The most severe problem with [unsafe function-like macros](/sei-cert-c-coding-standard/back-matter/bb-definitions#BB.Definitions-unsafefunction-likemacro) is side effects of macro arguments, as shown in this noncompliant code example:

::code-block{quality="bad"}
``` c
#define ABS(x) (((x) < 0) ? -(x) : (x))

void f(int n) {
  int m;
  m = ABS(++n);
  /* ... */
}
```
::

The invocation of the `ABS()` macro in this noncompliant code example expands to the following code. The resulting code has well-defined behavior but causes `n` to be incremented twice rather than once, which may be surprising to those unfamiliar with the [implementation](/sei-cert-c-coding-standard/back-matter/bb-definitions#BB.Definitions-implementation) of the macro or unaware that they are using a macro in the first place.

::code-block{quality="bad"}
``` c
m = (((++n) < 0) ? -(++n) : (++n));
```
::

<a id="PRE12C.Donotdefineunsafemacros-cs_inline_function"></a>

## Compliant Solution (Inline Function)

A possible and preferable compliant solution is to define an inline function with equivalent but unsurprising semantics:

::code-block{quality="good"}
``` c
inline int Abs(int x) {
  return x < 0 ? -x : x;
}
```
::

<a id="PRE12C.Donotdefineunsafemacros-cs_language_extension"></a>

## Compliant Solution (Language Extension)

Some implementations provide language extensions that make it possible to define safe function-like macros, such as the macro `ABS()` , that would otherwise require evaluating their arguments more than once. For example, the GCC extension [Statements and Declarations in Expressions](https://gcc.gnu.org/onlinedocs/gcc/Statement-Exprs.html#Statement-Exprs) makes it possible to implement the macro `ABS()` in a safe way. Note, however, that because relying on implementation-defined extensions introduces undesirable platform dependencies that may make the resulting code nonportable, such solutions should be avoided in favor of portable ones wherever possible (see [MSC14-C. Do not introduce unnecessary platform dependencies](/sei-cert-c-coding-standard/recommendations/miscellaneous-msc/msc14-c) ).

Another GCC extension known as *statement expression* makes it possible for the block statement to appear where an expression is expected. The statement expression extension establishes a scope (note the curly braces) and any declarations in it are distinct from those in enclosing scopes.

  

::code-block{quality="good"}
``` c
#define ABS(x) __extension__ ({ __typeof (x) __tmp = x; __tmp < 0 ? - __tmp : __tmp; })
```
::

## Risk Assessment

Defining an unsafe macro leads to invocations of the macro with an argument that has [side effects](/sei-cert-c-coding-standard/back-matter/bb-definitions#BB.Definitions-sideeffect) , causing those side effects to occur more than once. [Unexpected](/sei-cert-c-coding-standard/back-matter/bb-definitions#BB.Definitions-unexpectedbehavior) or [undefined](/sei-cert-c-coding-standard/back-matter/bb-definitions#BB.Definitions-undefinedbehavior) program behavior can result.
