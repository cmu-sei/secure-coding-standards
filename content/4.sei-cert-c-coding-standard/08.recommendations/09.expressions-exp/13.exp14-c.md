# EXP14-C. Beware of integer promotion when performing bitwise operations on integer types smaller than int

Deprecated

This guideline has been deprecated by

-   [INT02-C. Understand integer conversion rules](/sei-cert-c-coding-standard/recommendations/integers-int/int02-c)

Integer types smaller than `int` are promoted when an operation is performed on them. If all values of the original type can be represented as an `int` , the value of the smaller type is converted to an `int` ; otherwise, it is converted to an `   unsigned int  ` (see [INT02-C. Understand integer conversion rules](/sei-cert-c-coding-standard/recommendations/integers-int/int02-c) ). If the conversion is to a wider type, the original value is zero-extended for unsigned values or sign-extended for signed types. Consequently, bitwise operations on integer types smaller than `int` may have unexpected results.

## Noncompliant Code Example

This noncompliant code example demonstrates how performing bitwise operations on integer types smaller than `int` may have unexpected results.

::code-block{quality="bad"}
``` c
uint8_t port = 0x5a;
uint8_t result_8 = ( ~port ) >> 4;
```
::

In this example, a bitwise complement of `port` is first computed and then shifted 4 bits to the right. If both of these operations are performed on an 8-bit unsigned integer, then `result_8` will have the value `0x0a` . However, `port` is first promoted to a `   signed int  ` , with the following results (on a typical architecture where type `int` is 32 bits wide):

## Compliant Solution

In this compliant solution, the bitwise complement of `port` is converted back to 8 bits. Consequently, `result_8` is assigned the expected value of `0x0aU` .

::code-block{quality="good"}
``` c
uint8_t port = 0x5a;
uint8_t result_8 = (uint8_t) (~port) >> 4;
```
::
