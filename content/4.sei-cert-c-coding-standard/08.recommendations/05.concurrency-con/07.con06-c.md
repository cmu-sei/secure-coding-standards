# CON06-C. Ensure that every mutex outlives the data it protects

  

## \*\*\*\*\*\*\*\*\*\* Text below this note not yet converted from Java to C! \*\*\*\*\*\*\*\*\*\*\*\*

Programs must not use instance locks to protect static shared data because instance locks are ineffective when two or more instances of the class are created. Consequently, failure to use a static lock object leaves the shared state unprotected against concurrent access. Lock objects for classes that can interact with untrusted code must also be private and final, as shown in rule [LCK00-J. Use private final lock objects to synchronize classes that may interact with untrusted code](/sei-cert-oracle-coding-standard-for-java/rules/locking-lck/lck00-j) .

## Risk Assessment

Using an instance lock to protect static shared data can result in nondeterministic behavior.
